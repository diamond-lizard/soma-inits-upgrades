* Overview

=soma-inits-upgrades= automates the security review and upgrade planning
process for stale [[https://github.com/progfolio/elpaca][elpaca]] package pins in your Emacs configuration.

Given a JSON file listing packages with outdated pinned refs, the program:

1. Clones each repository and generates a diff between the pinned ref and the
   latest commit.
2. Extracts Emacs Lisp symbols used by your configuration.
3. Pauses at three LLM-assisted review steps per init file, where you paste a
   generated prompt into your LLM of choice and save its response.
4. Validates LLM output and builds a dependency graph across all packages.
5. Produces per-init-file upgrade reports and a cross-package summary.

Processing is fully resumable.  If you quit at any point, re-running the
command picks up exactly where you left off.

* Prerequisites

- Python 3.10 or later
- [[https://docs.astral.sh/uv/][uv]] (for dependency management)
- git 2.19 or later (for blobless clones)
- [[https://github.com/BurntSushi/ripgrep][ripgrep]] with PCRE2 support (for Emacs Lisp symbol search)

* Installation

Clone this repository and create the virtual environment:

#+begin_example
cd soma-inits-upgrades
uv venv
uv pip install -e '.[dev]'
#+end_example

* Input File

The program expects a JSON file with the following structure:

#+begin_example
{
  "results": [
    {
      "init_file": "soma-magit-init.el",
      "repo_url": "https://github.com/magit/magit",
      "pinned_ref": "abc123def456..."
    },
    ...
  ]
}
#+end_example

Each entry describes one elpaca-pinned package:

- =init_file= -- the Emacs init file containing the pin for this package (e.g.
  =soma-magit-init.el=).
- =repo_url= -- the HTTPS URL of the package repository.
- =pinned_ref= -- the git commit hash currently pinned.

When an init file pins multiple packages from different repositories,
list each package as a separate entry with the same =init_file= value:

#+begin_example
{
  "results": [
    {
      "init_file": "soma-outshine-and-outorg-init.el",
      "repo_url": "https://github.com/alphapapa/outshine",
      "pinned_ref": "aaa111..."
    },
    {
      "init_file": "soma-outshine-and-outorg-init.el",
      "repo_url": "https://github.com/alphapapa/outorg",
      "pinned_ref": "bbb222..."
    }
  ]
}
#+end_example

Entries sharing the same =init_file= are automatically grouped and
processed as a single unit: per-repository operations (clone, diff,
symbol extraction) run independently for each repository, while
LLM-assisted reviews and reports are produced once covering all
repositories together.  The =(init_file, repo_url)= pair must be
unique across all entries.

* Usage

#+begin_example
soma-inits-upgrades STALE_INITS_FILE [--output-dir DIR]
#+end_example

- =STALE_INITS_FILE= -- path to the JSON input file.
- =--output-dir= -- where to write reports and state (default:
  =~/.emacs.d/soma/inits-upgrades/=).

On first run, the program prompts for your Emacs version.  This is used
to detect packages that require a newer Emacs than you have.

All files and directories that the program creates, deletes, or modifies
are confined to the output directory.  No files under =~/.emacs.d/= or
anywhere else on the filesystem are touched.  The input JSON file is
read but never modified.

* Workflow

Processing follows a two-tier model.  Tier 1 operations run
independently for each repository.  Once all repositories in an init
file complete Tier 1, Tier 2 operations run once per init file,
covering all its repositories together.  This means all automated
per-repository work finishes before the interactive LLM steps begin.

** Tier 1: Per-repository operations

*** Clone and diff

The repository is cloned (using blobless partial clones to save bandwidth),
the default branch and latest commit are identified, and a diff is generated
between the pinned ref and the latest commit.  If the pinned ref is already
at the latest commit, the repository is marked as current and skipped.

*** Dependency and symbol extraction

Package dependencies (from Emacs Lisp headers) are parsed.  If the package
requires a newer Emacs version than yours, it is flagged.  Emacs Lisp
symbols defined by the package are extracted, and your configuration is
searched to find which symbols you actually use.

After all Tier 1 tasks complete for a repository, its clone directory is
deleted to free disk space.  Diff and analysis files are retained for
Tier 2.

** Tier 2: Per-init-file operations

*** Security review (LLM-assisted)

A prompt file is generated containing the diffs and context for all
repositories in the init file.  The program pauses and asks you to:

1. Copy the prompt (optionally to the X primary selection via =xclip=).
2. Paste it into your LLM.
3. Save the LLM response to the indicated output file.
4. Press =(c)ontinue=, =(s)kip=, or =(q)uit=.

*** Upgrade analysis (LLM-assisted)

A second prompt is generated that asks the LLM to identify breaking API
changes, removed or renamed symbols, changed dependencies, and Emacs
version conflicts across all repositories in the init file.  The same
pause-and-paste workflow applies.  The LLM response must be valid JSON
matching the expected schema.

*** Upgrade report (LLM-assisted)

A third prompt asks the LLM to produce a human-readable upgrade report
covering all repositories in the init file.  Same pause-and-paste
workflow.

*** Graph update and validation

Dependency information for all packages in the init file is recorded in
a cross-package dependency graph.  After all init files are processed,
the graph is validated for cycles.

** Cleanup

Remaining temporary artifacts (diffs, analysis files, prompt copies) are
deleted after Tier 2 completes.

* Output

All output is written under the output directory (default
=~/.emacs.d/soma/inits-upgrades/=):

| File                                   | Description                              |
|----------------------------------------+------------------------------------------|
| =<init-file>-security-review.md=      | LLM security review per init file        |
| =<init-file>-upgrade-process.md=      | LLM upgrade report per init file         |
| =soma-inits-dependency-graphs.json=   | Cross-package dependency graph            |
| =security-review-summary.md=          | Aggregated security review summary        |
| =emacs-version-conflicts.md=          | Packages requiring a newer Emacs          |
| =dependency-graph-warnings.md=        | Dependency graph validation warnings      |

Security reviews and upgrade reports are produced once per init file,
even when the init file contains multiple packages from different
repositories.  Each entry in the dependency graph lists all packages
belonging to its init file, with per-package dependency and version
information.

Intermediate files (diffs, symbol data, prompt copies) are organized
under =.tmp/{init-file}/= with per-repository subdirectories.  State
tracking files are stored under =.state/=.  Both are managed
automatically and should not be edited by hand.

* Resumability

The program tracks progress in JSON state files under =.state/=.  You can
quit at any time (=q= at a prompt, =Ctrl-C=, or =SIGTERM=) and re-run the
same command to resume.

If the input file changes between runs (e.g. a =repo_url= or =pinned_ref=
is updated, or a repository is added to or removed from an init file),
the affected init files are automatically reset and reprocessed.  New
init files are picked up; init files no longer in the input are orphaned
and their graph entries removed.

Entries that encounter errors are retried automatically (up to five
times) on subsequent runs.

* Process Locking

Only one instance of the program may run against a given output directory
at a time.  A lock file under =.state/= prevents concurrent runs.
